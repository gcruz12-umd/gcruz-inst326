Why Code Reuse Matters

Imagine you’re building a social media app with different post types:

class Tweet:
    def __init__(self, creator, content):
        self.creator = creator
        self.content = content
        self.likes = 0

    def like(self):
        self.likes += 1

    def delete(self):
        print(f"Deleted by {self.creator}")

class InstagramPost:
    def __init__(self, creator, content):
        self.creator = creator
        self.content = content
        self.likes = 0

    def like(self):
        self.likes += 1

    def delete(self):
        print(f"Deleted by {self.creator}")

Problem: Most of this code is duplicated!

This is a common situation. You have similar classes with overlapping functionality.

Copy-pasting leads to: - More code to maintain - Bugs fixed in one place but not another - Inconsistent behavior over time

We need better patterns for sharing code.

Inheritance

What Is Inheritance?

Inheritance lets a class reuse code from another class.

class Post:           # Parent class (superclass)
    def __init__(self, creator):
        self.creator = creator

    def delete(self):
        print(f"Deleted by {self.creator}")

class Tweet(Post):    # Child class (subclass)
    pass              # Inherits everything from Post!

The child class (Tweet) automatically gets all the methods and attributes of the parent class (Post).

The syntax class Tweet(Post): declares that Tweet inherits from Post.

Even with pass (no new code), Tweet can do everything Post can do.

How Inheritance Works

class Post:
    def __init__(self, creator):
        self.creator = creator

    def delete(self):
        print(f"Deleted by {self.creator}")

class Tweet(Post):
    def retweet(self):
        print(f"Retweeted by {self.creator}")

class Snap(Post):
    def expire(self):
        print(f"Snap by {self.creator} expired")
tweet = Tweet("alice")
tweet.delete()      # Inherited from Post!
tweet.retweet()     # Defined in Tweet

snap = Snap("bob")
snap.delete()       # Inherited from Post!
snap.expire()       # Defined in Snap

Key points: - Both Tweet and Snap inherit init and delete() from Post - Each subclass can add its own methods - The parent’s code is written once, used by all children

Turn to a neighbor and discuss for 60 seconds.

AttributeError: 'Tweet' object has no attribute 'expire'

Siblings don’t share methods—only parent methods are inherited.

This is a common misconception. Tweet and Snap are siblings (both inherit from Post), but they don’t share methods with each other.

Tweet only has access to: - Methods defined in Post (inherited) - Methods defined in Tweet itself

Not methods from Snap or any other sibling.

type() vs. isinstance()

class Post:
    pass

class Tweet(Post):
    pass

tweet = Tweet()
type(tweet) == Tweet    # True - exact type
type(tweet) == Post     # False - not exactly a Post

isinstance(tweet, Tweet)  # True - is a Tweet
isinstance(tweet, Post)   # True - is ALSO a Post!

An instance of a subclass is also an instance of the parent class.

This is important for understanding the "is-a" relationship.

A Tweet IS a Post. It has all the capabilities of a Post.

type() checks the exact class. isinstance() checks if an object is an instance of a class OR any of its parent classes.

Use isinstance() when you care about capabilities, not exact type.

The Liskov Substitution Principle

Barbara Liskov

MIT Professor, first American woman to earn a Ph.D. in computer science

The Principle:

If S is a subclass of C, then instances of C may be replaced with instances of S without breaking the program.

In other words: A subclass should be usable anywhere its parent class is expected.

This is a fundamental principle of good object-oriented design.

If your code expects a Post, it should work correctly with a Tweet (since Tweet is a Post).

Violating this principle leads to bugs and surprises. If a subclass doesn’t behave like its parent, something is wrong with your design.

Overriding Methods

Why Override Methods?

Sometimes a subclass needs to change how a method works, not just add new methods.

class Post:
    def __init__(self, creator):
        self.creator = creator

    def edit(self, new_content):
        self.content = new_content
        print("Post edited")

class Tweet(Post):
    def edit(self, new_content):
        # Tweets can't be edited!
        print("Error: Tweets cannot be edited")

Overriding: Defining a method in the subclass with the same name as a parent method.

The subclass method "shadows" the parent method. When you call tweet.edit(), Python uses the Tweet version, not the Post version.

This is how subclasses customize behavior while maintaining the same interface.

Overriding: A First Attempt

class BankAccount:
    def __init__(self, owner, balance=0):
        self.owner = owner
        self.balance = balance

    def withdraw(self, amount):
        if amount <= self.balance:
            self.balance -= amount
            print(f"Withdrew ${amount}. Balance: ${self.balance}")

class SavingsAccount(BankAccount):
    def withdraw(self, amount):
        if amount <= self.balance:
            self.balance -= amount
            print(f"Withdrew ${amount}. Balance: ${self.balance}")
        if self.balance < 100:
            print(f"Warning: balance below $100 minimum!")

Problem: We copied all the code from BankAccount.withdraw() just to add a warning!

This defeats the purpose of inheritance. We’re duplicating code.

What we really want is to: 1. Do everything the parent method does 2. Add our extra behavior

We need a way to call the parent method from within our override.

Calling the Parent Method Directly

class BankAccount:
    def __init__(self, owner, balance=0):
        self.owner = owner
        self.balance = balance

    def withdraw(self, amount):
        if amount <= self.balance:
            self.balance -= amount
            print(f"Withdrew ${amount}. Balance: ${self.balance}")

class SavingsAccount(BankAccount):
    def withdraw(self, amount):
        BankAccount.withdraw(self, amount)  # Call parent method
        if self.balance < 100:
            print(f"Warning: balance below $100 minimum!")

Better! But we have to know the parent class name and pass self explicitly.

This works, but it’s clunky: - We hardcode the parent class name (BankAccount) - We have to pass self explicitly - If we change the parent class, we have to update all these calls

There’s a better way…​

super() to the Rescue

class BankAccount:
    def __init__(self, owner, balance=0):
        self.owner = owner
        self.balance = balance

    def withdraw(self, amount):
        if amount <= self.balance:
            self.balance -= amount
            print(f"Withdrew ${amount}. Balance: ${self.balance}")

class SavingsAccount(BankAccount):
    def withdraw(self, amount):
        super().withdraw(amount)  # Much cleaner!
        if self.balance < 100:
            print(f"Warning: balance below $100 minimum!")

super() returns a proxy to the parent class—no need to name it or pass self.

super() is the Pythonic way to call parent methods.

Benefits: - Don’t need to hardcode parent class name - Don’t need to pass self - Works correctly with multiple inheritance (advanced topic)

Use super() whenever you need to call a parent method from an override.

Common Pattern: Extending init

class Post:
    def __init__(self, creator):
        self.creator = creator
        self.likes = 0

class Tweet(Post):
    def __init__(self, creator, max_length=280):
        super().__init__(creator)  # Call parent __init__
        self.max_length = max_length  # Add new attribute
tweet = Tweet("alice", max_length=140)
print(tweet.creator)      # "alice" (from parent)
print(tweet.likes)        # 0 (from parent)
print(tweet.max_length)   # 140 (from Tweet)

Always call super().init() when overriding init!

This is one of the most common uses of super().

The pattern: 1. Call super().init() to set up parent attributes 2. Then set up child-specific attributes

If you forget to call super().init(), the parent’s attributes won’t be initialized, leading to AttributeError later.

🔮 Predict: Method Resolution

class Animal:
    def speak(self):
        return "..."

    def describe(self):
        return f"I say {self.speak()}"

class Dog(Animal):
    def speak(self):
        return "Woof!"

dog = Dog()
print(dog.describe())  # What prints?

Turn to a neighbor and discuss for 60 seconds.

I say Woof!

Even though describe() is defined in Animal, it calls self.speak(), which uses Dog’s version!

This is called "dynamic dispatch" or "polymorphism."

When Animal.describe() calls self.speak(), Python looks up speak() on the actual object (a Dog), not the class where describe() is defined.

This is powerful—parent methods can use child overrides.

Abstract Base Classes

Sometimes we want to define an interface without implementing it:

class Player:
    """Abstract base class for game players."""

    def __init__(self, name):
        self.name = name

    def get_move(self, board):
        raise NotImplementedError("Subclasses must implement get_move")

class HumanPlayer(Player):
    def get_move(self, board):
        return input("Enter your move: ")

class ComputerPlayer(Player):
    def get_move(self, board):
        # AI logic here
        return self.calculate_best_move(board)

The parent defines WHAT methods exist; children define HOW they work.

Abstract base classes are useful when: - You need several classes with the same interface - The implementation differs between classes - You want to enforce that subclasses implement certain methods

NotImplementedError signals that this method must be overridden.

Both HumanPlayer and ComputerPlayer are "Players" and can be used interchangeably.

Inheritance: Summary

  • Subclasses inherit all methods and attributes from parent classes

  • Use class Child(Parent): to declare inheritance

  • Override methods to change behavior

  • Use super() to call parent methods from overrides

  • Always call super().init() when overriding init

  • An instance of a subclass IS an instance of the parent class

Composition

What Is Composition?

Composition means objects contain references to other objects.

Instead of "is-a" (inheritance), composition models "has-a" relationships.

A Pizza HAS Toppings. A Car HAS an Engine. A University HAS Students.

The containing object uses the contained objects to do work.

This is a very common and flexible pattern.

Composition in Action

class Topping:
    def __init__(self, name, num_pieces):
        self.name = name
        self.num_pieces = num_pieces

    def __repr__(self):
        return f"{self.num_pieces} pieces of {self.name}"

class Pizza:
    def __init__(self):
        self.toppings = []  # Pizza HAS toppings

    def add_topping(self, topping):
        self.toppings.append(topping)
pizza = Pizza()
pizza.add_topping(Topping("pepperoni", 18))
pizza.add_topping(Topping("mushrooms", 12))
print(pizza.toppings)
# [18 pieces of pepperoni, 12 pieces of mushrooms]

Pizza doesn’t inherit from Topping. Instead, Pizza contains a list of Topping objects.

The relationship is "has-a": A pizza HAS toppings.

Each class is independent and focused on one thing.

Composition: Objects Working Together

class Topping:
    def __init__(self, name, num_pieces):
        self.name = name
        self.num_pieces = num_pieces

class Pizza:
    def __init__(self):
        self.toppings = []

    def add_topping(self, topping):
        self.toppings.append(topping)

    def total_pieces(self):
        """Count total topping pieces on the pizza."""
        count = 0
        for topping in self.toppings:
            count += topping.num_pieces
        return count
pizza = Pizza()
pizza.add_topping(Topping("pepperoni", 18))
pizza.add_topping(Topping("mushrooms", 12))
print(pizza.total_pieces())  # 30

The Pizza object "reads" data from its contained Topping objects.

This is composition in action—objects collaborating to achieve functionality.

Pizza doesn’t need to know how Topping works internally, just its interface (num_pieces).

Real-World Composition Example

class Engine:
    def __init__(self, horsepower):
        self.horsepower = horsepower
        self.running = False

    def start(self):
        self.running = True

    def stop(self):
        self.running = False

class Car:
    def __init__(self, make, model, horsepower):
        self.make = make
        self.model = model
        self.engine = Engine(horsepower)  # Car HAS an Engine

    def start(self):
        self.engine.start()
        print(f"{self.make} {self.model} started!")
car = Car("Toyota", "Camry", 200)
car.start()  # "Toyota Camry started!"

A Car HAS an Engine. The Car delegates engine-related work to the Engine object.

This is cleaner than having Car inherit from Engine (a Car is NOT an Engine).

Each class has a clear, focused responsibility.

🔮 Predict: Composition Behavior

class Battery:
    def __init__(self, capacity):
        self.capacity = capacity
        self.charge = capacity

    def drain(self, amount):
        self.charge = max(0, self.charge - amount)

class Phone:
    def __init__(self):
        self.battery = Battery(100)

    def use(self, minutes):
        self.battery.drain(minutes * 2)

phone = Phone()
phone.use(30)
print(phone.battery.charge)  # What prints?

Turn to a neighbor and discuss for 60 seconds.

40 (Started at 100, drained 30 * 2 = 60)

Walk through the logic: 1. Phone creates a Battery with capacity 100, charge starts at 100 2. phone.use(30) calls battery.drain(60) 3. drain sets charge to max(0, 100 - 60) = 40

Composition: Summary

  • Composition models "has-a" relationships

  • Objects contain references to other objects

  • Each class has a focused responsibility

  • Objects collaborate by calling each other’s methods

  • More flexible than inheritance—can change components at runtime

Inheritance vs. Composition

When to Use Inheritance

Use inheritance when:

  • There’s a clear "is-a" relationship

  • Subclasses genuinely ARE specialized versions of the parent

  • You need polymorphism (treat children as parent type)

  • There’s significant shared behavior

# Good use of inheritance
class Animal:
    def eat(self): ...

class Dog(Animal):    # A Dog IS an Animal
    def bark(self): ...

class Cat(Animal):    # A Cat IS an Animal
    def meow(self): ...

Inheritance is best when subclasses really are specialized versions of the parent.

Dogs and Cats are both Animals. They share eating behavior and can be treated as Animals.

When to Use Composition

Use composition when:

  • There’s a "has-a" or "uses-a" relationship

  • You want loose coupling between classes

  • Components might be shared or swapped

  • You’re combining unrelated behaviors

# Good use of composition
class Car:
    def __init__(self):
        self.engine = Engine()    # Car HAS an Engine
        self.wheels = [Wheel() for _ in range(4)]

class Motorcycle:
    def __init__(self):
        self.engine = Engine()    # Motorcycle also HAS an Engine
        self.wheels = [Wheel() for _ in range(2)]

Both Car and Motorcycle have engines, but neither IS an engine.

With composition, they can share the Engine class without being related by inheritance.

You could even swap engines at runtime!

Common Mistake: Inheritance Abuse

# BAD: Using inheritance for code reuse when there's no "is-a"
class Database:
    def query(self, sql): ...

class UserManager(Database):  # UserManager IS a Database? No!
    def create_user(self, name): ...
# GOOD: Use composition instead
class Database:
    def query(self, sql): ...

class UserManager:
    def __init__(self, database):
        self.db = database  # UserManager HAS a Database

    def create_user(self, name):
        self.db.query(f"INSERT INTO users ...")

Ask yourself: Is this really an "is-a" relationship?

A UserManager is NOT a Database. It USES a database.

Using inheritance here is awkward and limits flexibility. What if you want to use different database backends?

With composition, you can pass in any database object.

The Design Patterns Perspective

"Favor composition over inheritance."

— Gang of Four, Design Patterns (1994)

Composition is often more flexible because:

  • Components can be swapped at runtime

  • Classes are more loosely coupled

  • Changes to one class don’t ripple to others

This is famous advice from one of the most influential programming books.

It doesn’t mean "never use inheritance"—it means default to composition and use inheritance when it genuinely fits.

Inheritance creates tight coupling. Changes to the parent affect all children.

Quick Decision Guide

Should I use inheritance or composition?

1. Is there a clear "is-a" relationship?
   - Yes: Consider inheritance
   - No: Use composition

2. Do I need polymorphism (treat objects as parent type)?
   - Yes: Consider inheritance
   - No: Use composition

3. Will the relationship change at runtime?
   - Yes: Use composition
   - No: Either could work

4. Am I just trying to reuse some code?
   - Code reuse alone isn't enough for inheritance
   - Use composition or other patterns

This is a simplified guide. Real designs often involve both patterns.

The key insight: Inheritance is about IS-A relationships and polymorphism, not just code reuse.

Practice Challenge: Solution (Part 1 - Inheritance)

class Character:
    def __init__(self, name, health=100):
        self.name = name
        self.health = health
        self.inventory = Inventory()  # Composition!

    def take_damage(self, amount):
        self.health -= amount
        if self.health < 0:
            self.health = 0
        print(f"{self.name} takes {amount} damage! Health: {self.health}")

    def is_alive(self):
        return self.health > 0

class Warrior(Character):
    def __init__(self, name, health=100):
        super().__init__(name, health)  # Call parent __init__
        self.rage = 0

    def berserk(self):
        self.rage += 10
        print(f"{self.name} goes berserk! Rage: {self.rage}")

class Mage(Character):
    def __init__(self, name, health=80, mana=100):
        super().__init__(name, health)  # Call parent __init__
        self.mana = mana

    def cast_spell(self, spell_name, mana_cost=10):

            self.mana -= mana_cost

Practice Challenge: Solution (Part 2 - Composition)

class Inventory:
    def __init__(self):
        self.items = []

    def add_item(self, item):
        self.items.append(item)
        print(f"Added {item.name} to inventory")

    def remove_item(self, item_name):
        for item in self.items:
            if item.name == item_name:
                self.items.remove(item)
                print(f"Removed {item_name} from inventory")
                return item
        print(f"{item_name} not found in inventory")
        return None

    def list_items(self):
        if not self.items:
            print("Inventory is empty")
        else:
            print("Inventory:")
            for item in self.items:
                print(f"  - {item}")
