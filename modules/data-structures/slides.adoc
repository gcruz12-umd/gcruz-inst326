== Why Different Data Structures?

Each data structure has *strengths* and *trade-offs*:

[cols="1,2,2", frame=none]
|===
| Structure | Good For | Trade-off

| List | Ordered, changeable collections | Slow membership testing
| Tuple | Immutable sequences, dict keys | Can't modify
| Set | Fast membership, uniqueness | No order, no duplicates
| Dictionary | Key-value lookups | Keys must be hashable
|===

*Choosing the right structure makes your code faster and clearer.*

[.notes]
--
A common beginner mistake is using lists for everything. Lists are versatile but not always optimal.

Example: Checking if an item is in a list of 1 million items is slow. Checking if it's in a set is nearly instant.

We'll explore each structure's sweet spot.

== Tuples

=== What Is a Tuple?

A tuple is an *immutable* ordered sequence.

[source,python]
----
# Creating tuples
point = (3, 4)
person = ("Alice", 25, "Engineer")
singleton = (42,)      # Note the comma!
empty = ()

# Accessing elements (same as lists)
x = point[0]           # 3
name = person[0]       # "Alice"
----

*Like a list, but you can't change it after creation.*

[.notes]
--
Tuples use parentheses instead of square brackets.

The comma is what makes a tuple, not the parentheses. `(42)` is just the number 42 in parentheses. `(42,)` is a tuple containing 42.

Why use tuples? They're:
- Safer (can't accidentally modify)
- Hashable (can be dictionary keys)
- Slightly more memory-efficient
--

=== Tuple vs. List: The Key Difference

[source,python]
----
# Lists are mutable
my_list = [1, 2, 3]
my_list[0] = 99        # Works!

# Tuples are immutable
my_tuple = (1, 2, 3)
my_tuple[0] = 99       # TypeError!
----

*Once created, a tuple cannot be changed.*

This immutability makes tuples:

* Safer (data won't change unexpectedly)
* Hashable (usable as dictionary keys)
* A signal of intent ("this data is fixed")

[.notes]
--
This is the fundamental difference. Lists can be modified in place; tuples cannot.

This immutability is actually useful:
- Guarantees data won't change unexpectedly
- Makes tuples hashable (usable as dict keys)
- Signals intent: "This data is fixed"
--

=== Comparing Tuples to Other Sequences

[cols="1,1,1,1", frame=none]
|===
| Feature | Tuple | List | String

| Ordered | Yes | Yes | Yes
| Indexed | Yes | Yes | Yes
| Sliceable | Yes | Yes | Yes
| Mutable | *No* | Yes | No
| Hashable | Yes | *No* | Yes
| Element types | Any | Any | Characters only
|===

[.notes]
--
Tuples share many features with lists and strings—they're all sequences.

The key differences:
- Tuples are immutable like strings, but can hold any type like lists
- Tuples are hashable, which means they can be dictionary keys (lists cannot)
--

=== Tuple Packing and Unpacking

[source,python]
----
# Packing: multiple values into one tuple
coordinates = 3, 4, 5    # Parentheses optional

# Unpacking: tuple into multiple variables
x, y, z = coordinates    # x=3, y=4, z=5

# Swap values elegantly
a, b = b, a              # No temp variable needed!

# Ignore values with underscore
first, _, third = (1, 2, 3)   # Ignore the middle value
----

[.notes]
--
Unpacking is one of the most useful Python features.

The swap idiom `a, b = b, a` works because Python evaluates the right side first (creating a tuple), then unpacks it to the left side.

Using `_` for ignored values is a convention—it's a valid variable name, but signals "I don't care about this value."
--

=== Creating and Returning Tuples from Functions

[source,python]
----
def min_max(numbers):
    """Return both minimum and maximum."""
    return min(numbers), max(numbers)

# Calling it
data = [3, 1, 4, 1, 5, 9, 2, 6]
smallest, largest = min_max(data)
print(f"Range: {smallest} to {largest}")
----

Output: `Range: 1 to 9`

*Functions can return tuples to give back multiple values.*

[.notes]
--
This is extremely common in Python. When a function needs to return multiple pieces of information, return a tuple and let the caller unpack it.

Built-in examples:
- `divmod(7, 3)` returns `(2, 1)` (quotient and remainder)
- `dict.items()` returns tuples of (key, value)
--

=== More Examples: Returning Tuples

[source,python]
----
def analyze_text(text):
    """Return word count, character count, and average word length."""
    words = text.split()
    word_count = len(words)
    char_count = len(text)
    avg_length = sum(len(w) for w in words) / word_count if words else 0
    return word_count, char_count, avg_length

# Unpack the returned tuple
words, chars, avg = analyze_text("Hello world from Python")
print(f"Words: {words}, Chars: {chars}, Avg length: {avg:.2f}")
----

Output: `Words: 4, Chars: 23, Avg length: 5.00`

[.notes]
--
Returning tuples is a clean way to return related pieces of information without creating a class or dictionary.

The caller can unpack into meaningfully named variables, making the code readable.
--

=== When to Use Tuples

Use tuples when:

* Data shouldn't change (coordinates, RGB colors)
* Returning multiple values from a function
* Using as dictionary keys (lists can't be keys!)
* Unpacking in `for` loops: `for name, age in people:`

[source,python]
----
# Tuples as dictionary keys
locations = {
    (40.7128, -74.0060): "New York",
    (51.5074, -0.1278): "London"
}

# Iterating with unpacking
for coords, city in locations.items():
    print(f"{city} is at {coords}")
----

[.notes]
--
The dictionary key example is important. You CAN'T use lists as dictionary keys because they're mutable. Tuples work because they're immutable.

The iteration example shows tuple unpacking in a for loop—very common when processing structured data.
--

=== Tuples: Summary

* Tuples are *immutable* ordered sequences
* Created with `()` or just commas: `x, y, z`
* Support indexing, slicing, `count()`, `index()`
* *Cannot* be modified after creation
* Great for multiple return values and dict keys
* Hashable (unlike lists)

== Sets

=== What Is a Set?

A set is an *unordered* collection of *unique* elements.

[source,python]
----
# Creating sets
colors = {"red", "green", "blue"}
numbers = {1, 2, 3, 2, 1}    # Duplicates removed!
print(numbers)               # {1, 2, 3}

# From other sequences
letters = set("hello")       # {'h', 'e', 'l', 'o'}

# Empty set (NOT {}, that's an empty dict!)
empty = set()
----

[.notes]
--
Sets come from mathematics. Key properties:
- No duplicates (each element appears once)
- No order (can't index with [0])
- Very fast membership testing

The `set("hello")` example shows automatic deduplication—'l' only appears once.

IMPORTANT: `{}` creates an empty dictionary, NOT an empty set! Use `set()` for empty sets.
--

=== Sets vs. Other Sequence Types

[cols="1,1,1,1", frame=none]
|===
| Feature | Set | List | Tuple

| Ordered | *No* | Yes | Yes
| Indexed | *No* | Yes | Yes
| Duplicates | *No* | Yes | Yes
| Mutable | Yes | Yes | No
| Fast `in` | *Yes* | No | No
|===

[.notes]
--
Sets are fundamentally different from lists and tuples:
- No order means you can't access by index
- No duplicates means each element appears exactly once
- Fast membership testing is the key advantage

Sets are optimized for answering "is X in this collection?" quickly.
--

=== Common Mistake: Empty Set vs. Empty Dict

[source,python]
----
# This creates an empty DICTIONARY, not a set!
wrong = {}
print(type(wrong))    # <class 'dict'>

# This creates an empty SET
correct = set()
print(type(correct))  # <class 'set'>
----

*Use `set()` to create an empty set. `{}` is always an empty dictionary.*

[.notes]
--
This is a historical quirk of Python. Dictionaries came first and got the `{}` syntax. Sets were added later.

When you see `{}`, think "dictionary" unless there are elements inside without colons.

`{1, 2, 3}` is a set. `{1: "a", 2: "b"}` is a dictionary.
--

=== Use Cases for Sets

Sets are ideal when you need to:

* *Remove duplicates* from a collection
* *Test membership* quickly ("Is X in this collection?")
* *Find common elements* between collections
* *Find differences* between collections

[source,python]
----
# Remove duplicates from a list
names = ["Alice", "Bob", "Alice", "Carol", "Bob"]
unique_names = list(set(names))  # ["Alice", "Bob", "Carol"]

# Fast membership testing
valid_users = {"alice", "bob", "carol"}
if username.lower() in valid_users:
    print("Access granted")
----

[.notes]
--
These are the four main use cases for sets.

Converting a list to a set and back is the fastest way to remove duplicates (though it loses order).

For membership testing, sets are dramatically faster than lists for large collections.
--

=== Why Sets Are Fast

Sets use *hashing* for near-instant lookups.

[source,python]
----
# Checking membership in a list: O(n) - slow!
big_list = list(range(1000000))
999999 in big_list    # Checks each element

# Checking membership in a set: O(1) - fast!
big_set = set(range(1000000))
999999 in big_set     # Direct lookup via hash
----

*For large collections, sets can be 1000x faster for membership testing.*

[.notes]
--
The `in` operator is the key operation.

For a list, Python must potentially check every element. For a set, Python computes a hash and jumps directly to the location.

If you're checking membership frequently, convert to a set first!
--

=== Set Operations: Union

*Union* combines all elements from both sets.

[source,python]
----
a = {1, 2, 3, 4}
b = {3, 4, 5, 6}

# Union: all elements from both sets
a | b              # {1, 2, 3, 4, 5, 6}
a.union(b)         # Same thing
----

image::set_union.png[Set Union, 300]

Think: "Everything in either A *or* B (or both)"

[.notes]
--
Union (|): "Everything in either set"

The method name (`union`) and operator (`|`) do the same thing. Use whichever is clearer in context.

Union creates a new set; it doesn't modify the originals.
--

=== Set Operations: Intersection

*Intersection* finds elements common to both sets.

[source,python]
----
a = {1, 2, 3, 4}
b = {3, 4, 5, 6}

# Intersection: elements in both sets
a & b              # {3, 4}
a.intersection(b)  # Same thing
----

image::set_intersection.png[Set Intersection, 300]

Think: "Only things in *both* A *and* B"

[.notes]
--
Intersection (&): "Only things in both sets"

This is useful for finding common elements between two collections.
--

=== Set Operations: Difference

*Difference* finds elements in one set but not the other.

[source,python]
----
a = {1, 2, 3, 4}
b = {3, 4, 5, 6}

# Difference: elements in a but not in b
a - b              # {1, 2}
a.difference(b)    # Same thing

# Note: order matters!
b - a              # {5, 6}
----

image::set_difference.png[Set Difference, 300]

Think: "Things in A that *aren't* in B"

[.notes]
--
Difference (-): "Things in the first set that aren't in the second"

Important: `a - b` is different from `b - a`! Order matters for difference.
--

=== Set Operations: Symmetric Difference

*Symmetric difference* finds elements in either set, but not both.

[source,python]
----
a = {1, 2, 3, 4}
b = {3, 4, 5, 6}

# Symmetric difference: in either but not both
a ^ b              # {1, 2, 5, 6}
a.symmetric_difference(b)
----

image::set_symmetric_diff.png[Symmetric Difference, 300]

Think: "Things in A *or* B, but *not* in both"

[.notes]
--
Symmetric difference (^): "Things in one or the other, but not both"

This is like XOR for sets—it excludes the overlap.
--

=== Building Sets with Operations

[source,python]
----
# Start with one set
languages = {"Python", "Java"}

# Add elements
languages.add("JavaScript")
# {"Python", "Java", "JavaScript"}

# Add multiple elements (union in place)
languages.update(["Ruby", "Go"])
# {"Python", "Java", "JavaScript", "Ruby", "Go"}

# Remove elements
languages.remove("Java")      # Raises KeyError if not found
languages.discard("Rust")     # No error if not found
----

[.notes]
--
Unlike tuples, sets ARE mutable. You can add and remove elements.

`remove()` vs `discard()`: Use `remove()` when the element should exist (error = bug). Use `discard()` when it might not exist (error = annoying).
--

== Lists as Stacks and Queues

=== What Are Stacks and Queues?

Two fundamental ways to organize data:

[cols="1,1", frame=none]
|===
| *Stack* | *Queue*

| Last In, First Out (LIFO)
| First In, First Out (FIFO)

| Like a stack of plates
| Like a line at a store

| Add/remove from *top* only
| Add at *back*, remove from *front*
|===

[.notes]
--
These are abstract data structures that describe how items are added and removed.

Stack: Think of a stack of plates—you add to the top and take from the top. The last plate you put on is the first one you take off.

Queue: Think of a line at a store—first person in line is first to be served.
--

=== Stacks: Last In, First Out (LIFO)

[source,python]
----
# Using a list as a stack
stack = []

# Push: add to top
stack.append("first")
stack.append("second")
stack.append("third")
print(stack)          # ["first", "second", "third"]

# Pop: remove from top
top = stack.pop()     # "third"
print(stack)          # ["first", "second"]
----

*`append()` pushes, `pop()` pops—both operate on the end.*

[.notes]
--
Lists make natural stacks because `append()` and `pop()` both work on the end of the list, which is efficient.

The key insight: the last item added is the first one removed.
--

=== Stack Visualization

[source]
----
Operation         Stack State
---------         -----------
stack = []        []

append("A")       ["A"]
                      ↑ top

append("B")       ["A", "B"]
                           ↑ top

append("C")       ["A", "B", "C"]
                                ↑ top

pop() → "C"       ["A", "B"]
                          ↑ top

pop() → "B"       ["A"]
                      ↑ top
----

[.notes]
--
Walk through this visualization:
- Items are always added to and removed from the top (right side)
- "C" was added last, so it's removed first
- "A" was added first, so it would be removed last
--

=== Use Cases for Stacks

* *Undo functionality* — Each action goes on the stack; undo pops the most recent
* *Browser history* — Back button pops the most recent page
* *Function call stack* — Python tracks function calls this way
* *Expression evaluation* — Parsing mathematical expressions

[source,python]
----
# Simple undo system
actions = []

def do_action(action):
    actions.append(action)
    print(f"Did: {action}")

def undo():
    if actions:
        undone = actions.pop()
        print(f"Undone: {undone}")
----

[.notes]
--
Stacks are used constantly in programming:
- Your editor's undo is a stack
- When Python calls a function, it pushes info onto the call stack
- When the function returns, it pops from the stack

This is why you see "stack trace" in error messages!
--

=== Queues: First In, First Out (FIFO)

[source,python]
----
from collections import deque

# Using deque as a queue (more efficient than list)
queue = deque()

# Enqueue: add to back
queue.append("first")
queue.append("second")
queue.append("third")
print(queue)          # deque(['first', 'second', 'third'])

# Dequeue: remove from front
front = queue.popleft()  # "first"
print(queue)             # deque(['second', 'third'])
----

*`append()` adds to back, `popleft()` removes from front.*

[.notes]
--
For queues, we use `collections.deque` (double-ended queue) because removing from the front of a list is slow (everything must shift).

`deque` is optimized for fast operations at both ends.
--

=== Why Use deque for Queues?

[source,python]
----
# Using a list as a queue (SLOW!)
queue = []
queue.append("a")
queue.pop(0)        # O(n) - must shift all elements!

# Using deque as a queue (FAST!)
from collections import deque
queue = deque()
queue.append("a")
queue.popleft()     # O(1) - no shifting needed!
----

*`list.pop(0)` is O(n) because all elements must shift.*

*`deque.popleft()` is O(1) because it's designed for this.*

[.notes]
--
This is an important efficiency consideration.

When you remove from the front of a list, every other element must shift left to fill the gap. For a list of 1 million items, that's 1 million shifts!

Deque uses a different internal structure that allows fast operations at both ends.
--

=== Queue Visualization

[source]
----
Operation          Queue State
---------          -----------
queue = deque()    []

append("A")        [A]
                   front → A ← back

append("B")        [A, B]
                   front → A   B ← back

append("C")        [A, B, C]
                   front → A   B   C ← back

popleft() → "A"    [B, C]
                   front → B   C ← back

popleft() → "B"    [C]
                   front → C ← back
----

[.notes]
--
Walk through this visualization:
- Items enter at the back (right side)
- Items exit from the front (left side)
- "A" was added first, so it's removed first
- This is like a line at a store
--

=== Use Cases for Queues

* *Task scheduling* — Process jobs in the order received
* *Print queue* — Documents print in submission order
* *Message queues* — Handle messages in arrival order
* *Breadth-first search* — Graph traversal algorithm

[source,python]
----
from collections import deque

# Simple task processor
tasks = deque()

def add_task(task):
    tasks.append(task)
    print(f"Added: {task}")

def process_next():
    if tasks:
        task = tasks.popleft()
        print(f"Processing: {task}")
----

[.notes]
--
Queues ensure fairness—first come, first served.

This is why your operating system uses queues for scheduling tasks, and printers use queues for print jobs.

== List Comprehensions

=== What Is a List Comprehension?

A *concise* way to create lists from other sequences.

[source,python]
----
# Traditional approach
squares = []
for x in range(10):
    squares.append(x ** 2)

# List comprehension (same result!)
squares = [x ** 2 for x in range(10)]
----

Both produce: `[0, 1, 4, 9, 16, 25, 36, 49, 64, 81]`

*One line instead of three!*

[.notes]
--
List comprehensions are a Python idiom for creating lists concisely.

They're not just shorter—they're often faster because Python can optimize them internally.

Read it as: "Create a list of x squared, for each x in range(10)"
--

=== Anatomy of a List Comprehension

[source]
----
[expression for item in iterable]
----

[source,python]
----
[x ** 2 for x in range(10)]
 ↑       ↑    ↑
 |       |    └── Source of items
 |       └─────── Loop variable
 └─────────────── What to put in the list
----

[.notes]
--
Break down the structure:
1. Expression: What to compute for each item
2. for item in iterable: Where the items come from
3. Square brackets: This creates a list

The expression can be any valid Python expression using the loop variable.
--

=== Adding Conditions (Filtering)

[source,python]
----
# Only include even numbers
evens = [x for x in range(10) if x % 2 == 0]
# [0, 2, 4, 6, 8]

# Only include words that start with 'a'
words = ["apple", "banana", "apricot", "cherry"]
a_words = [w for w in words if w.startswith("a")]
# ["apple", "apricot"]
----

*Add `if condition` at the end to filter.*

[.notes]
--
The `if` clause acts as a filter—only items that pass the condition are included.

Read it as: "Create a list of x, for each x in range(10), if x is even"

This replaces the pattern:
```python
result = []
for x in items:
    if condition:
        result.append(x)
```
--

=== Transforming and Filtering Together

[source,python]
----
# Get lengths of words longer than 3 characters

== Dictionaries

=== Dictionary Recap

Dictionaries store *key-value pairs* for fast lookup.

[source,python]
----
# Creating dictionaries
person = {"name": "Alice", "age": 25}
empty = {}
also_empty = dict()

# Accessing values

----

*Keys must be hashable (immutable). Values can be anything.*

[.notes]
--
Quick recap of dictionary basics.

The `.get()` method is safer than `[]` because it doesn't raise KeyError for missing keys.

=== Valid Dictionary Keys

*Keys must be hashable* (immutable types):

[cols="1,1", frame=none]
|===
| Valid Keys | Invalid Keys

| Strings: `"name"`
| Lists: `[1, 2, 3]`

| Numbers: `42`, `3.14`
| Dictionaries: `{"a": 1}`

| Tuples: `(1, 2)`
| Sets: `{1, 2, 3}`

| Booleans: `True`, `False`
| Any mutable type
|===

[source,python]
----
valid = {(1, 2): "point", "name": "test", 42: "answer"}

invalid = {[1, 2]: "list"}  # TypeError: unhashable type: 'list'
----

[.notes]
--
Why must keys be hashable? Dictionaries use hashing internally to look up values quickly. If a key could change, its hash would change, and the dictionary couldn't find it anymore.

This is the same reason lists can't be set elements or dictionary keys.

[.notes]

While keys must be hashable, values have no restrictions. You can store lists, other dictionaries, functions—anything.

Nested dictionaries are common for representing structured data like JSON.

=== Counting with Dictionaries

One of the most common patterns: counting occurrences.

[source,python]
----
# Count word frequencies
text = "the cat sat on the mat"
word_counts = {}

for word in text.split():
    if word in word_counts:
        word_counts[word] += 1
    else:
        word_counts[word] = 1

print(word_counts)
# {'the': 2, 'cat': 1, 'sat': 1, 'on': 1, 'mat': 1}
----

[.notes]
--
This is the basic counting pattern:
1. Check if the key exists
2. If yes, increment the count
3. If no, initialize to 1

This pattern is so common that Python provides better ways to do it...
--

=== Counting: Using .get() Method

[source,python]
----
# Cleaner counting with .get()
text = "the cat sat on the mat"
word_counts = {}

for word in text.split():
    word_counts[word] = word_counts.get(word, 0) + 1

print(word_counts)
# {'the': 2, 'cat': 1, 'sat': 1, 'on': 1, 'mat': 1}
----

*`.get(key, default)` returns the default if key doesn't exist.*

[.notes]
--
This is cleaner because we don't need the if/else.

`.get(word, 0)` returns 0 if the word isn't in the dictionary yet.

Then we add 1 and store the result.
--

=== Counting: Using collections.Counter

*`Counter` is a specialized dictionary for counting.*

[.notes]

`Counter` is purpose-built for counting. It:
- Automatically counts elements
- Provides `most_common()` method
- Supports arithmetic operations between counters

For counting tasks, `Counter` is usually the best choice.

[source,python]

== How Hash Maps Work

=== What Is a Hash Map?

A *hash map* (also called a *hash table*) is the data structure underlying Python dictionaries and sets.

*Key features:*

* Stores key-value pairs
* Allows *fast* access, insertion, and deletion
* Uses a *hash function* to determine where to store each item

*Python's `dict` is a hash map.*

[.notes]
--
Understanding how hash maps work helps you:
- Understand why keys must be hashable
- Know when dictionaries are fast vs slow
- Debug unexpected behavior

Let's build one from scratch to see how it works.
--

=== Hash Functions

A *hash function* converts a key into an array index.

[source,python]
----
# Python's built-in hash function
hash("apple")      # Some large integer like 8729374928374
hash("banana")     # Different integer
hash(42)           # 42 (small integers hash to themselves)

# To get an array index, use modulo
size = 5
index = hash("apple") % size   # 0, 1, 2, 3, or 4
----

*The same key always produces the same hash.*

[.notes]
--
The hash function is the key to fast lookups.

Instead of searching through all items, we:
1. Compute the hash of the key
2. Use modulo to get an index in our array
3. Go directly to that index

This is O(1) instead of O(n)!
--

=== Hash Map Structure

A hash map is an array of *buckets*. Each bucket can hold key-value pairs.

[source]
----
Hash Table with 3 buckets:

Index 0: [  ]  ← empty bucket
Index 1: [  ]  ← empty bucket  
Index 2: [  ]  ← empty bucket
----

[source,python]
----
class HashTable:
    def __init__(self, size):
        self.size = size
        self.hash_table = [[] for _ in range(size)]
----

[.notes]
--
We create an array of empty lists (buckets).

The size determines how many buckets we have. More buckets = fewer collisions, but more memory.

Each bucket is a list that can hold multiple items (for handling collisions).
--

=== Inserting: Step by Step

Let's insert `"apple": 10` into a hash table of size 3.

[source]
----
Step 1: Compute hash
   hash("apple") = 8729374928374 (example)

Step 2: Get bucket index
   8729374928374 % 3 = 1

Step 3: Store in bucket 1
   Index 0: [  ]
   Index 1: [("apple", 10)]  ← stored here!
   Index 2: [  ]
----

[source,python]
----
def set_val(self, key, val):
    hashed_key = hash(key) % self.size
    bucket = self.hash_table[hashed_key]
    bucket.append((key, val))
----

[.notes]
--
Walk through each step:
1. Hash the key to get a large integer
2. Use modulo to get an index within our array
3. Add the key-value pair to that bucket

This is the basic insertion. We'll refine it to handle updates.
--

=== Inserting Multiple Items

[source]
----
Insert "apple": 10    (hash % 3 = 1)
Insert "banana": 20   (hash % 3 = 2)
Insert "cherry": 30   (hash % 3 = 0)

Result:
   Index 0: [("cherry", 30)]
   Index 1: [("apple", 10)]
   Index 2: [("banana", 20)]
----

Each item lands in a different bucket based on its hash.

[.notes]
--
When different keys hash to different indices, each bucket has at most one item.

Lookup is fast: hash the key, go directly to the bucket, return the value.

But what happens when two keys hash to the same index?
--

=== Collisions: When Keys Hash to the Same Index

[source]
----
Insert "apple": 10    (hash % 3 = 1)
Insert "date": 40     (hash % 3 = 1)  ← Same bucket!

Result:
   Index 0: [("cherry", 30)]
   Index 1: [("apple", 10), ("date", 40)]  ← Two items!
   Index 2: [("banana", 20)]
----

*Collision:* Two keys map to the same bucket.

*Solution:* Store multiple items in each bucket (chaining).

[.notes]
--
Collisions are inevitable—we're mapping infinite possible keys to a finite number of buckets.

"Chaining" means we store a list at each bucket. When there's a collision, we add to the list.

To find an item, we hash to the bucket, then search the list.
--

=== Handling Collisions in Code

[source,python]
----
def set_val(self, key, val):
    hashed_key = hash(key) % self.size
    bucket = self.hash_table[hashed_key]
    
    # Check if key already exists (update)
    for index, (record_key, _) in enumerate(bucket):
        if record_key == key:
            bucket[index] = (key, val)  # Update existing
            return
    
    # Key doesn't exist, add new
    bucket.append((key, val))
----

[.notes]
--
The full insertion logic:
1. Find the bucket
2. Search the bucket for the key
3. If found, update the value
4. If not found, append a new pair

This handles both new insertions and updates.
--

=== Retrieving Values

[source]
----
Get value for "date":
   Step 1: hash("date") % 3 = 1
   Step 2: Go to bucket 1
   Step 3: Search bucket: [("apple", 10), ("date", 40)]
   Step 4: Found "date" → return 40
----

[source,python]
----
def get_val(self, key):
    hashed_key = hash(key) % self.size
    bucket = self.hash_table[hashed_key]
    
    for record_key, record_val in bucket:
        if record_key == key:
            return record_val
    
    return "No record found"
----

[.notes]
--
Retrieval is similar:
1. Hash to find the bucket
2. Search the bucket for the key
3. Return the value if found

If there are few collisions, the bucket is small and search is fast.
--

=== Deleting Values

[source]
----
Delete "apple":
   Step 1: hash("apple") % 3 = 1
   Step 2: Go to bucket 1: [("apple", 10), ("date", 40)]
   Step 3: Find "apple", remove it
   Step 4: Bucket now: [("date", 40)]
----

[source,python]
----
def delete_val(self, key):
    hashed_key = hash(key) % self.size
    bucket = self.hash_table[hashed_key]
    
    for index, (record_key, _) in enumerate(bucket):
        if record_key == key:
            bucket.pop(index)
            return
----

[.notes]
--
Deletion:
1. Hash to find the bucket
2. Search for the key
3. Remove the item from the bucket

Note: We don't need to shift other items between buckets.
--

=== Complete HashTable Implementation

[source,python]
----
class HashTable:
    def __init__(self, size):
        self.size = size
        self.hash_table = [[] for _ in range(size)]

    def set_val(self, key, val):
        hashed_key = hash(key) % self.size
        bucket = self.hash_table[hashed_key]
        for index, (record_key, _) in enumerate(bucket):
            if record_key == key:
                bucket[index] = (key, val)
                return
        bucket.append((key, val))

    def get_val(self, key):
        hashed_key = hash(key) % self.size
        bucket = self.hash_table[hashed_key]
        for record_key, record_val in bucket:
            if record_key == key:
                return record_val
        return "No record found"

    def delete_val(self, key):
        hashed_key = hash(key) % self.size
        bucket = self.hash_table[hashed_key]
        for index, (record_key, _) in enumerate(bucket):
            if record_key == key:
                bucket.pop(index)
                return
----

[.notes]
--
This is a complete working hash table.

Python's built-in dict is more sophisticated (dynamic resizing, optimizations), but the core idea is the same.
--

[source,python]
----

ht.set_val('apple', 10)
ht.set_val('banana', 20)
ht.set_val('cherry', 30)

print("Value for 'banana':", ht.get_val('banana'))  # 20
print("Value for 'apple':", ht.get_val('apple'))    # 10

ht.set_val('apple', 50)  # Update existing key
print("Updated apple:", ht.get_val('apple'))        # 50

ht.delete_val('banana')
print("After deletion:", ht.get_val('banana'))      # No record found
----

[.notes]
--
Walk through each operation:
- Insert three items
- Retrieve values (direct lookup)
- Update an existing key
- Delete a key and verify it's gone
--

=== Hash Map Performance

[cols="1,1,1", frame=none]
|===
| Operation | Average | Worst Case

| Insert | O(1) | O(n)
| Lookup | O(1) | O(n)
| Delete | O(1) | O(n)
|===

*Average case is constant time!* 

Worst case happens when all keys hash to the same bucket (everything in one list).

[.notes]
--
This is why dictionaries are so fast—most operations are O(1).

The worst case is rare with a good hash function and enough buckets.

Python's dict handles this by resizing when the table gets too full.
--

=== Why Keys Must Be Hashable

[source,python]
----
# This works
d = {(1, 2): "tuple key"}

# This fails!
d = {[1, 2]: "list key"}  # TypeError: unhashable type: 'list'
----

*If a key could change, its hash would change, and we couldn't find it!*

[source,python]
----
# Imagine if this worked (it doesn't!)
key = [1, 2]
d = {key: "value"}    # Stored at hash([1, 2]) % size
key.append(3)         # Now key is [1, 2, 3]
# d[key] would look at hash([1, 2, 3]) % size — wrong bucket!
----

[.notes]
--
This is the deep reason why dictionary keys must be immutable.

The hash of a key determines where it's stored. If the key changes, the hash changes, and we can't find the value anymore.

Tuples are immutable, so their hash never changes. Lists can change, so they can't be hashed reliably.
--

=== Hash Maps: Summary

* Hash maps provide *fast key-value storage* (O(1) average)
* *Hash function* converts keys to array indices
* *Collisions* handled by chaining (lists in buckets)
* Keys must be *hashable* (immutable) so their hash doesn't change
* Python's `dict` and `set` are hash maps internally

from collections import Counter, defaultdict

def analyze_words(filename):
    """Analyze words in a text file."""
    with open(filename, "r", encoding="utf-8") as f:
        text = f.read().lower()
    
    words = text.split()
    
    # 1. Unique words (set)
    unique = set(words)
    
    # 2. Word frequency (Counter)
    counts = Counter(words)
    
    # 3. Index by first letter (defaultdict)
    by_letter = defaultdict(list)
    for word in unique:  # Use unique to avoid duplicates in lists

    # 4. Top 5 most common
    top_5 = counts.most_common(5)
    
    return unique, counts, dict(by_letter), top_5
